# Implementation Plan: Hospital Platform Full-Stack Enhancements

## Background & Motivation
The Fettle hospital platform requires a comprehensive set of UI/UX improvements, functional additions to the PWA transcription app, and new backend automation logic for reminder calls and prescription notifications. The goal is to improve the usability of the dashboard, provide more detailed analytics, streamline WhatsApp campaigns, and fully automate the post-consultation workflow.

## Scope & Impact
- **Frontend (`frontend_fettel`)**: Redesigning top KPI cards, fixing Engagement and Intent distribution graphs, overhauling the Campaign Management (Agent Control) UI, adding Presigned S3 file uploads for WhatsApp templates, detailing ROI formulas, adding an Inbound/Outbound toggle, updating Staff Management to use mobile numbers and availability schedules, and displaying Digital Prescriptions in the Transcripts tab.
- **Backend (`fettle_backend`)**: Providing Presigned S3 URLs, fixing PDF metrics reports, updating Staff models for availability, and implementing a Celery Beat task for automated 24hr and 1hr reminder calls based on revisit dates.
- **PWA (`pwa-plan`)**: Updating the Medivoice interface to capture diagnosis, medicines, revisit date/time, and triggering post-confirmation emails (reception/pharmacy) and WhatsApp messages (patient) with the generated prescription PDF.

## Proposed Solution
- **Reminder Infrastructure**: Use Celery Beat (Periodic Polling) to scan `MediVoiceSession` every 15 minutes for appointments exactly 24 hours and 1 hour away, dispatching outbound voice agents.
- **Media Uploads**: Use Direct S3 Upload (Presigned URLs) generated by Django. The frontend will request a URL, upload the video/image directly to AWS S3, and pass the final object URL to the WhatsApp campaign endpoint.

## Alternatives Considered
- **Exact Scheduling (Celery ETA)**: Using `apply_async(eta=...)` for reminders was rejected due to difficulties in canceling or modifying tasks if an appointment changes.
- **Django Pass-through Uploads**: Routing media files through Django via `multipart/form-data` was rejected to avoid unnecessary load on the Python server for large video files.

## Implementation Phases

### Phase 1: Dashboard UI & Engagement Analytics
1.  **Top Cards**: Refactor `AdminOverview.tsx` to expand the 4 KPI cards (Total Patients, Call Answer Rate, etc.) to full viewport width.
2.  **Inbound/Outbound Toggle**: Add a global toggle in the header (`Index.tsx` or `AdminDashboard.tsx`) to filter all dashboard analytics by direction.
3.  **Engagement Graphs**: Fix `CommunityEngagementMetrics.tsx` and `PatientEngagementMetrics.tsx`. Implement the "Feedback Distribution" bar chart (clickable to show patient details) and populate the "Patient Intents" distribution chart.
4.  **ROI Details**: Add detailed formulas and calculation breakdowns in `ROIAnalytics.tsx` for monetary savings and operational efficiency.
5.  **Terminology Update**: Replace "Launch Robot" with "Start Hospital Outreach" across all components.

### Phase 2: Campaign Management & WhatsApp Media (Frontend & Backend)
1.  **Agent Command Center**: Redesign `AgentControl.tsx`. Clarify the audience selection flow (Upload New Excel vs Historical Date Range vs All Patients) to distinguish campaign triggers from general list calling.
2.  **S3 Presigned URLs (Backend)**: Create an endpoint in Django `utils/s3_uploader.py` and `views.py` to generate presigned URLs for direct uploads.
3.  **WhatsApp Templates (Frontend)**: Update `UploadSection.tsx` to support native picture/video uploads via S3 presigned URLs. Ensure submitting a template explicitly prompts the user to upload an Excel sheet for targeted delivery.

### Phase 3: Staff Availability & Transcripts (Frontend & Backend)
1.  **Staff Model (Backend)**: Update `Doctor_model` (or equivalent staff model) to include `mobile_number` and `availability_schedule` (JSON field).
2.  **Staff UI**: Update `StaffManagement.tsx` to use mobile numbers instead of emails and build a dynamic schedule selector tab for days/timings.
3.  **Voice Agent Context**: Ensure staff availability is injected into the metadata of the `call_outbound_task` so the LiveKit agent knows when doctors are free for scheduling.
4.  **Transcript PDF Display**: Update `DoctorTranscripts.tsx` to reliably fetch and render the digital prescription alongside the AI summary.

### Phase 4: Reports & PWA Integration
1.  **PDF Reports (Backend)**: Fix the `download_excel_outbound` or related metrics endpoints to output PDFs instead of `.docx` and ensure the summary logic differs from the detailed report.
2.  **PWA Fields (`pwa-plan`)**: Update Medivoice schemas (`draftConsultationDocumentSchema`) and UI components to include `diagnosis`, `medicines`, `revisit_date`, and `revisit_time`. Ensure `syncWithFettle` pushes these to Django.
3.  **Post-Confirmation Triggers (`pwa-plan`)**: Update `pdf-delivery-service.ts` to automatically send emails to the reception/pharmacy and a WhatsApp message with the PDF to the patient once the doctor confirms the prescription.

### Phase 5: Automated Reminder Calls (Backend)
1.  **Celery Beat Task**: Create a periodic task in `phone_calling/tasks.py` and schedule it in `project/celery.py`.
2.  **Polling Logic**: Query `MediVoiceSession` where `revisit_date` and `revisit_time` fall within the 24-hour or 1-hour window.
3.  **Call Dispatch**: Initiate outbound calls via the LiveKit SIP bridge using `call_outbound_task`, passing the appointment context.

## Verification & Testing
-   **Unit & Integration**: Run `npm run lint` and `pytest` locally for frontend and backend changes.
-   **E2E Testing**: Log in to `hospital.fettleconnect.com` with `Amor`/`Amor` credentials. Upload a CSV to trigger an outbound call to `+919003037804` exclusively from the frontend GUI. Verify the call connects and the agent speaks.
-   **PWA Sync Check**: Generate a prescription in the PWA, confirm it, and verify that emails/WhatsApp messages are sent and the data appears in the Fettle Transcripts tab.

## Migration & Rollback
-   Database migrations are required for the Staff availability fields.
-   Changes will be pushed to isolated feature branches (e.g., `feature/hospital-enhancements`) and PRs raised.
-   Rollback involves reverting to `master`/`main` and reversing the Django migration.
